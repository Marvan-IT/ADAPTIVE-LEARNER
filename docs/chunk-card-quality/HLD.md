# HLD — Chunk Card Quality Fix

## 1. Executive Summary

**Feature name:** Chunk Card Quality Fix

**Purpose:** Resolve four defects in the chunk-based card generation path (`generate_per_chunk`) that cause cards to arrive in the wrong structural format, without images, with too little content due to token starvation, and generated by an under-powered model.

**Business problem:** Students enrolled in the chunk architecture learning path receive lesson cards that alternate between content-only and MCQ-only cards (breaking the combined content+MCQ contract), rarely show relevant diagrams, and are thin in explanation because the token budget allows only ~230 tokens per card. All four issues degrade learning quality and contradict the adaptive teaching design.

**Key stakeholders:** Learners (primary), backend developer, curriculum QA.

**Scope:**
- Included: prompt instruction correction, image injection fallback, token budget constants, LLM model selection for `generate_per_chunk`.
- Excluded: frontend changes, database schema changes, Alembic migrations, other card generation paths (`generate_per_card`, `_generate_cards_single`).

---

## 2. Functional Requirements

| ID | Requirement |
|----|-------------|
| FR-1 | Every card produced by `generate_per_chunk` MUST contain both a `content` field and a `question` field (COMBINED card). |
| FR-2 | Every card that corresponds to a chunk with associated images MUST attempt to assign `image_url`; `null` is only acceptable when no images exist for the chunk. |
| FR-3 | Backend MUST inject `image_url` from the chunk image list when the LLM leaves the field null and images are available. |
| FR-4 | The token budget for NORMAL mode MUST be sufficient to generate 4–6 full content+MCQ cards (minimum 5000 tokens). |
| FR-5 | `generate_per_chunk` MUST use `gpt-4o` (main model) for primary and retry calls. |

---

## 3. Non-Functional Requirements

| Category | Target |
|----------|--------|
| Latency | NORMAL mode p50 latency increase acceptable up to +2 s vs current (gpt-4o is slower than gpt-4o-mini) |
| Cost | ~3–4× token cost per chunk call vs current mini usage; acceptable given quality improvement |
| Correctness | Zero combined-card violations in output (FR-1); image assignment rate ≥ 80% for chunks with images |
| Backwards compatibility | No change to public API contracts or DB schema |
| Rollback | Single constant change reverts model; single constant change reverts token budget |

## 5. Architectural Style and Patterns

The fix is a **surgical patch** to an existing service-layer function and its prompt builder. No new components, no new layers. All four changes are in-place edits to three files. This is consistent with all prior bug-fix work in this project (see `docs/pre-deployment-audit-fixes/`).

## 7. Key Architectural Decisions (ADRs)

### ADR-1: Fix prompt instruction rather than override system prompt in caller
**Options:** (a) Fix the user prompt in `build_chunk_card_prompt()`. (b) Add a post-processing normalisation step that splits alternating cards into combined ones.

**Chosen:** Option (a).

**Rationale:** Option (b) would require understanding which alternating-pair belongs together and re-assembling content — brittle and lossy. The root cause is a contradictory instruction; fixing the instruction at the source is correct and minimal.

---

### ADR-2: Backend image fallback rather than prompt-only fix
**Options:** (a) Strengthen prompt language to "MUST set image_url". (b) Strengthen prompt AND add backend fallback injection.

**Chosen:** Option (b).

**Rationale:** LLMs are probabilistic. Even with a "MUST" instruction the model may still return null for some cards. A deterministic backend fallback guarantees images appear when they are available, independent of model behaviour. Both layers are cheap to add.

---

### ADR-3: Raise NORMAL token budget to 5000
**Options:** Various values from 3000–8000.

**Chosen:** 5000.

**Rationale:** At ~400 tokens per combined card (content paragraph + MCQ + options + explanation), 5000 provides headroom for 8–10 cards. The NORMAL mode target is 4–6 cards per chunk. 5000 is conservative enough to stay within gpt-4o's context window cost envelope and fast enough to meet latency NFR.

---

### ADR-4: Switch to gpt-4o for generate_per_chunk
**Options:** (a) Keep gpt-4o-mini, fix prompt/tokens. (b) Switch to gpt-4o.

**Chosen:** Option (b).

**Rationale:** `generate_per_chunk` is called once per chunk (a coarse granularity, not per-card). The combined-card format, strict coverage rule, and MERGE RULES are complex multi-constraint instructions that gpt-4o-mini systematically fails to follow. The retry path also uses mini, compounding the failure rate. Switching both primary and retry calls to gpt-4o resolves instruction-following failures at the source.

---

## 8. Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| gpt-4o latency increase exceeds user tolerance | Low | p50 is ~3–4 s for 5000 tokens; still within acceptable range for a chunk load event |
| API cost increase surprises operators | Medium | Document in release notes; add `CHUNK_CARD_MODEL` config constant so operators can revert to mini if needed |
| Image fallback assigns wrong image to wrong card | Low | Fallback only fires when LLM returns null; first image from chunk is assigned — acceptable since chunk images are already filtered to educational relevance |
| Prompt change causes LLM to generate fewer cards | Low | System prompt COMBINED CARDS rule (line 1495 in teaching_service.py) is the authoritative instruction; user prompt change aligns with it |

---

## Key Decisions Requiring Stakeholder Input

1. **Cost acceptance:** Switching to gpt-4o approximately triples cost per chunk generation call. Confirm this is acceptable before shipping.
2. **Image fallback strategy:** The fallback assigns the first available chunk image to all image-null cards in that chunk. Confirm this "first image wins" behaviour is acceptable, or specify a preferred strategy (e.g., only the first card in the chunk gets the image).
