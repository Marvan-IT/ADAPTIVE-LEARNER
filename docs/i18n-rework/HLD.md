# HLD — i18n End-to-End Rework

**Feature slug:** `i18n-rework`
**Date authored:** 2026-04-23
**Status:** Ready for implementation

---

## 1. Executive Summary

The Adaptive Learner app ships with i18next resource bundles for 13 languages, but language switching is incomplete and slow. When a student switches from English to Malayalam (or any other language), the UI chrome changes immediately while data-bearing surfaces — concept map node titles, dashboard recent-activity titles, book/subject labels — remain in English. Worse, the language-change API endpoint blocks on a 2–5 s OpenAI call every time it fires. This rework fixes all five layers: persistent JSONB translation columns in the database, per-language card cache in `presentation_text`, deletion of the runtime LLM-translate path, correct re-fetch wiring in the frontend, and closure of 33 missing locale keys across 12 non-English files.

The business goal is that switching language translates **everything visible** within the time it takes the overlay to close. Post-fix the language-change endpoint must complete in p95 < 100 ms, and subsequent page navigations must show translated content without any additional LLM calls.

## 5. Data-Flow Walkthrough

### Scenario A: Student switches EN → ML on Dashboard

1. Student clicks Malayalam in `<LanguageSelector prominent />` on `DashboardPage.jsx`.
2. `selectLanguage()` shows the overlay immediately, then calls `updateLanguage(student.id, "ml")`.
3. `PATCH /students/{id}/language` runs in < 100 ms:
   - Updates `students.preferred_language = "ml"`.
   - Reads `heading_translations["ml"]` for active session's chunks (DB SELECT, no LLM).
   - Sets the `by_language` lazy-regen marker on the session cache (does not wipe EN cards).
   - Returns `translated_headings` (from DB) and `session_cache_cleared: true`.
4. On success: `i18n.changeLanguage("ml")` + `localStorage.setItem("ada_language", "ml")` + `dispatch(LANGUAGE_CHANGED, {headings})`.
5. `LANGUAGE_CHANGED` reducer updates chunk headings from the translated list; **preserves** `cardAnswers`, `currentCardIndex`, `maxReachedIndex`.
6. `i18n.language` dep triggers Dashboard's two `useEffect` hooks:
   - `getAvailableBooks()` → backend returns books with ML titles from `title_translations["ml"]`.
   - `getSessions(student.id)` → backend returns sessions with ML `concept_title` and `book_title`.
7. Dashboard re-renders fully in Malayalam. Overlay closes.

### Scenario B: Student starts a new lesson in ML for the first time

1. Student clicks a concept node on the concept map (loaded in ML).
2. `POST /api/v2/sessions` creates session; returns `SessionResponse` (concept/book titles not in this response).
3. `GET /api/v2/sessions/{id}/chunk-cards` is called (CARDS phase).
4. Backend reads `presentation_text` → no `by_language.ml` slice found.
5. `CacheAccessor.get_language_slice("ml")` returns `None`.
6. Backend calls `TeachingService` which generates ML cards via LLM (existing prompt engineering — `_language_instruction("ml")` appended).
7. On completion, `CacheAccessor.set_language_slice("ml", data)` writes the new slice. LRU cap enforced.
8. Next visit to the same concept in ML: cache hit — no LLM call.

### Scenario C: Operator adds a new book via pipeline

1. `python -m src.pipeline --book business_statistics --chunks`
2. Pipeline runs Stages 0–2 as before (Mathpix → parse → embed → DB persist).
3. After `graph.json` is written (end of Stage 2), pipeline calls `translate_catalog_inline("business_statistics")`.
4. The inline function fetches all rows needing translation (where `heading_translations` is `'{}'` or `en_source_hash` differs).
5. Batches headings into arrays of ≤ 50 items; calls OpenAI `gpt-4o-mini` with the array-in/array-out prompt.
6. On success: `UPDATE concept_chunks SET heading_translations = '{"en_source_hash":"<sha1>","ml":"...","ta":"...",...}'`.
7. Per-language failures log a warning; the row's entry for that language is left absent (English fallback at read time).
8. Also translates `books.title_translations`, `subjects.label_translations`, `chunk_images.caption_translations`.
9. Pipeline exits. Book is immediately available with translated titles for all served languages.

## 6. Architectural Style and Patterns

**Selected pattern:** Persistent translation columns with read-time overlay (the "Netflix/YouTube" approach for metadata translation).

**Why not runtime LLM translation:**
- 2–5 s latency on every language switch is unacceptable UX.
- Costs accumulate with each user interaction.
- Failures silently serve stale English content with no retry discipline.

**Why not a separate translations microservice:**
- The existing PostgreSQL schema can absorb JSONB columns with zero structural change.
- The team is Python/FastAPI/SQLAlchemy 2.0 — adding a new service would require new infra, new deployment artifacts, and new failure modes.

**Why JSONB instead of a translations table:**
- A `concept_chunk_translations` join table would require an extra join on every endpoint that already joins `concept_chunks`. JSONB colocates translations with the source row — one read, no join.
- JSONB supports partial updates (only add new language entry when available).
- Trade-off: JSONB is schema-free inside; the `{"en_source_hash": ..., "ml": ..., "ta": ...}` shape must be enforced by application code and documented.

**Why per-language slice inside `presentation_text` (not a new column):**
- `presentation_text` is already a `Text` column holding JSON; no migration needed.
- The per-language slice architecture is backward-compatible (legacy-shape adapter).
- Trade-off: 512 KB cap on the row must be monitored; telemetry alarm at p99 > 500 KB.

## 8. Key Architectural Decisions (ADRs)

### ADR-01: JSONB translation columns co-located with source rows

**Decision:** Add `*_translations JSONB` columns to `books`, `subjects`, `concept_chunks`, `chunk_images`.

**Options considered:**
- A. Separate `*_translations` join tables (one row per language per source row).
- B. JSONB column in the source table.
- C. External translation service (e.g., dedicated microservice or Redis hash).

**Chosen:** B.

**Rationale:** Option A doubles query complexity on all existing endpoints. Option C introduces a new deployment dependency. Option B is additive, colocated, and expressible in the same SQLAlchemy query path. Hash stored inside JSONB (`en_source_hash`) enables idempotent re-runs.

### ADR-02: Per-language slice inside `presentation_text`, no new DB column

**Decision:** Restructure `presentation_text` JSON shape; add legacy-shape adapter.

**Rationale:** `TeachingSession.presentation_text` is already a `Text` column; adding a new column would require modifying the ORM model, a new migration, and changes to every query that reads the session row. The adapter safely wraps legacy flat JSON on first read after deploy.

### ADR-03: `get_request_language()` — student profile wins for lesson content, header wins for catalog

**Decision:** One helper function encodes this rule; all endpoints use it.

**Rationale:** A logged-in student's `preferred_language` is the authoritative preference for LLM-generated lesson content (it was explicitly chosen). For stateless catalog endpoints (`/books`, `/graph/full`) the student may not be authenticated, so the `Accept-Language` header is the only signal. One helper, tested once, prevents per-endpoint drift.

### ADR-04: Inline synchronous translation at pipeline end (not async background job)

**Decision:** `translate_catalog_inline()` called synchronously at the end of `run_pipeline()`.

**Rationale:** Async background jobs require an external queue (Celery, Redis, etc.) — out of scope. Inline synchronous keeps failure visible in the operator's terminal; the pipeline already exits non-zero on stage errors. If translation fails, the book is still published (English fallback is safe), and the operator can re-run `scripts/translate_catalog.py` independently.

### ADR-05: Delete `POST /api/v2/concepts/translate-titles` in Phase 4 (not immediately)

**Decision:** Endpoint is removed only after Phase 3 confirms DB translations are populated and all callers are updated.

**Rationale:** Phase 3 is when the DB columns exist and are backfilled for Business Statistics. Frontend removes the `translateConceptTitles` call in the same phase. Phase 4 is the cleanup; deploying in sequence prevents a window where the frontend calls an endpoint that no longer exists.

## 9. Failure Modes and Mitigations

| Failure | Mitigation |
|---------|-----------|
| OpenAI outage during backfill | Per-language failures are logged as warnings; do not abort the batch. English fallback at read time. Operator re-runs `python scripts/translate_catalog.py --book <slug> --languages <failed-code>`. |
| `PATCH /students/{id}/language` DB timeout | Language update is a single row UPDATE; retry once. If still fails, return 503 — overlay shows error toast, language not changed. |
| `presentation_text` row exceeds 512 KB | LRU evicts oldest language slice before write. Telemetry counter `session_cache_eviction_total` fired; alert at > 10/hour. |
| `heading_translations[lang]` absent (untranslated book) | `resolve_translation()` returns English column value. Response contract: field is always non-empty. |
| Legacy `presentation_text` shape on deploy (flat JSON) | `CacheAccessor._ensure_by_language_shape()` wraps on first read; writes new shape back. Covered by `test_legacy_cache_shape_adapter.py`. |
| Frontend sends `Accept-Language: en-US` | `get_request_language()` normalizes to `"en"` via `split("-")[0]`. |
| Locale file with missing key after backfill | `npm run lint:locales` blocks the PR. Key falls back to `en` in i18next by default (safe). |
| Student resumes session in different language than it was created | `CacheAccessor` looks for the new language slice; missing → generates lazily. No progress loss. |
| JSONB `en_source_hash` collision (different English strings with same sha1) | SHA-1 collision probability is negligible for < 1 M short strings. If ever needed, upgrade to SHA-256 in the JSONB shape without DB schema change. |

## 10. Storage Analysis

| Data | Estimate | Bound |
|------|----------|-------|
| `concept_chunks.heading_translations` per book | ~50 bytes/heading × 13 languages × ~500 chunks = ~325 KB | < 1 MB |
| `books.title_translations` per book | ~60 bytes × 13 = < 1 KB | Negligible |
| `chunk_images.caption_translations` per book | ~100 bytes × 13 × ~200 images = ~260 KB | < 1 MB |
| `presentation_text` per session (per-language) | ~80 KB per language slice × 5 (LRU cap) = ~400 KB max | 512 KB hard cap |
| Total additional DB storage (2 books) | ~4 MB | Acceptable |

Cleanup crons (to be added as part of Phase 5):
- Sessions `completed_at IS NOT NULL AND completed_at < now() - interval '7 days'`: purge `presentation_text` column value only (keep row for history).
- Sessions with no `completed_at` and `started_at < now() - interval '90 days'`: full row delete.

## 11. Security and Privacy

- Translation columns contain textbook headings and book titles — no PII.
- OpenAI calls for translation use the same API key and scope as existing card generation.
- The backfill script writes directly to the PostgreSQL DB via the application's `DATABASE_URL` — must not be run with `--languages all` on a database with student PII unless that database connection is already permitted by the security policy.
- `resolve_translation()` reads from DB-controlled data; no user-supplied input reaches the translation resolution path.
- No change to authentication, JWT handling, or rate limiting.

## 12. Explicit Non-Goals

- **Prealgebra backfill** — explicitly deferred. Operator re-runs `scripts/translate_catalog.py --book prealgebra` when ready.
- **RTL layout** — Arabic is in the language list but RTL CSS is not part of this rework.
- **Student-editable translation corrections** — translations are operator-managed, not student-editable.
- **14th+ language support** — the design is forward-compatible (add to `LANGUAGE_NAMES`, add locale file, run backfill), but adding languages is not in scope.
- **Mid-session language change** — explicitly out of scope by UI design; the selector is removed from lesson routes.
- **Prealgebra concept map translation** — the English-fallback guarantee means Prealgebra continues to work in English for any language request. No degradation; just no translation.
